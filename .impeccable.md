## Design Context

_Provisional context inferred from repository docs and the current UI. Confirm with the product owner before treating these as final brand rules._

### Users
OpenBot appears to be used by developers, operators, or technically fluent individual owners who run and monitor a personal AI agent runtime.
They use the interface to inspect system health, review conversations and memory, manage tools and schedules, and adjust runtime settings.
The primary usage context appears desktop-first, with mobile acting as a secondary triage surface rather than the main operating environment.

### Brand Personality
Calm, technical, trustworthy.
The interface should help users feel in control of an active system, not dazzled by decoration or overwhelmed by telemetry.
Status communication should feel precise and operational rather than playful or consumer-marketing oriented.

### Aesthetic Direction
Light-first operator console with restrained depth, clear hierarchy, and strong status legibility.
The intended direction seems closer to an internal operations workspace than a promotional SaaS landing page.
Avoid generic AI-dashboard tropes: heroic blue gradients, interchangeable stat cards, and decorative chrome that competes with data.

### Design Principles
1. Prioritize operator clarity over visual flourish.
2. Make status, action hierarchy, and system state obvious within one screen.
3. Use visual emphasis sparingly so alerts and critical actions retain meaning.
4. Treat mobile as a triage experience that preserves access to core navigation and readable charts.
5. Prefer semantically rich, accessible controls over div-based approximations.
